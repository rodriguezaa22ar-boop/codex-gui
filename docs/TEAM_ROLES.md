# Codex Team Operating Model

This repo is built by a small distributed Codex team. GitHub `main` is the
source of truth. Devices do not improvise against stale checkouts; every worker
pulls first, works in a lane, commits focused changes, and hands results back to
the Ubuntu commander for integration.

## Device Roles

### `atlas-ubuntu`: Commander / Integrator

Purpose: keep the project coherent.

Responsibilities:

- Own `main`, GitHub sync, merge order, release notes, and conflict resolution.
- Write the mission packet before dispatching work.
- Review lane diffs before merging.
- Run the final quality gate before pushing.
- Keep a clear map of current blockers, device readiness, and next actions.

Boundaries:

- Do not disappear into a large feature lane.
- Do not merge unverified work just because it was generated by another agent.
- Prefer small integration commits with readable history.

Default commands:

```bash
git fetch origin main
git status --short --branch
python3 -m unittest discover -s tests
python3 -m py_compile codex_devices.py codex_gui.py codex_actions.py codex_team.py
git push origin main
```

### `atlas-builder`: Core Systems Engineer

Purpose: make the machinery reliable.

Responsibilities:

- Own mesh orchestration, Tailscale/SSH readiness, local/remote worker handling,
  config, persistence, command safety, packaging, and setup automation.
- Add tests for every backend behavior.
- Improve blocker diagnosis and repair suggestions.
- Keep remote command paths deterministic and metadata-safe.

Boundaries:

- Do not spend time on broad visual redesign.
- Do not change UX copy or layout except where backend state needs a control or
  status surface.
- Do not commit secrets, local device config, raw prompts, or terminal output.

Best current assignments:

- Add a mesh doctor CLI/report path.
- Harden worker setup for Ubuntu, NixOS, and clean clone laptops.
- Make SSH/Tailscale failures self-explanatory and actionable.
- Keep team sync, handoff bus, and collection behavior reversible.

### `atlas-main`: Product / GTK UX Engineer

Purpose: make the workstation feel like a serious product.

Responsibilities:

- Own the GUI experience: Mesh page, Command Palette, Quality Gate, Agent Team
  surfaces, visual hierarchy, text fit, interaction states, and workflow
  ergonomics.
- Make worker state obvious: ready, auth needed, offline, missing project,
  missing Codex, stale checkout, or blocked.
- Design dense operational surfaces rather than marketing pages.
- Keep controls discoverable and useful for repeated engineering work.

Boundaries:

- Do not rewrite backend behavior beyond small UI plumbing.
- Do not add decorative UI that hides status or slows the operator down.
- Keep text short enough to fit in panels, chips, buttons, and rows.

Best current assignments:

- Turn the Mesh page into an operations console.
- Add role badges, readiness timeline, and next-action buttons per device.
- Make the Command Palette expose the team workflow cleanly.
- Polish error, empty, loading, and blocked states.

### Fourth Laptop: Verifier / Release Engineer

Purpose: prove the project works away from the commander machine.

Responsibilities:

- Run fresh clone install checks.
- Confirm `codex-gui` launches from package install.
- Validate README, install docs, release checklist, and shell/Nix instructions.
- Run unit tests, compile checks, and public release checks.
- Capture exact failing commands and logs.

Boundaries:

- Do not make broad fixes first.
- Do not assume local commander state exists on a clean machine.
- Prefer precise failure reports and minimal patch recommendations.

Best current assignments:

- Clone from GitHub into a clean path.
- Run the install guide exactly as written.
- Verify the desktop app starts.
- Report missing packages, bad commands, and unclear docs.

Phones are out of scope for this project. They can approve Tailscale or read
GitHub, but they are not Codex workers for this build.

## Branch Rules

Use lane branches for worker changes:

```bash
lane/core-systems
lane/product-ux
lane/release-verify
```

The commander can commit directly to `main` only for integration, docs, release
notes, and small coordination changes. Worker devices should not free-edit
`main`.

## Standard Worker Loop

Every worker starts with:

```bash
cd ~/Projects/codex-gui
git fetch origin main
git checkout main
git pull --ff-only origin main
git checkout -B lane/<role-name>
```

Then:

1. Read this file and the current mission packet.
2. Make one scoped change.
3. Run the relevant checks.
4. Commit with a terse message.
5. Push the lane branch.
6. Write a handoff containing changed files, commands run, risks, and the next
   requested action.

## Integration Loop

Ubuntu integrates lanes in this order:

1. Pull or fetch each lane branch.
2. Review the diff by role boundary.
3. Run focused tests for that lane.
4. Merge one lane at a time.
5. Run the full suite.
6. Push `main`.
7. Tell every device to pull.

## Quality Gates

Minimum gate before pushing `main`:

```bash
python3 -m unittest discover -s tests
python3 -m py_compile codex_devices.py codex_gui.py codex_actions.py codex_team.py
```

When release or setup behavior changes, also run:

```bash
python3 -m unittest tests.test_setup tests.test_quality
```

When UI layout changes, the UX lane should include screenshot notes or visual
inspection notes in its handoff.

## Handoff Format

Each lane handoff should use this shape:

```text
Role:
Device:
Branch:
Commit:

Changed:
- file: reason

Validation:
- command: result

Risks:
- risk or none

Next:
- exact recommended next action
```

## Principle

The best version does not come from four devices coding randomly. It comes from
one commander, two builders, and one verifier working through GitHub with clear
ownership, explicit handoffs, and a quality gate that refuses vague success.
